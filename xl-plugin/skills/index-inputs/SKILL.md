# Index Input Policy Documents

Scan a domain's `input/policy_docs/` directory and produce:

- **`policy_facets/input-index.yaml`** — metadata only: the `files:` block (SHAs, md_quality scores). Always small. Fully rewritten on every run.
- **`policy_facets/computations/<rel>.md` (one file per source doc)** — per-file YAML lists of `{heading, summary, tags, computations}` blocks. Generated by the `/extract-computations` skill, which is invoked per REINDEX file. Sync state for these files lives in `policy_facets/.computations-manifest.yaml`.

On first run (CREATE), all files are indexed and every per-file computations file is generated. On subsequent runs (UPDATE), only changed or new files are re-processed using git SHA comparison; unchanged files are left alone via the computations manifest.

## Input

```
/index-inputs <domain>
```

If `<domain>` is not provided, list all `$DOMAINS_DIR/*/input/policy_docs/` directories as a numbered menu, prompt the user to choose, await their response, and use it as `<domain>` before continuing.

Read `../../core/output-fencing.md` now.

## Pre-flight

Run these checks before doing anything else:

1. **Domain argument provided?**
   - NO → List all directories matching `$DOMAINS_DIR/*/input/policy_docs/` as a numbered menu and prompt:
     :::user_input
     Available domains:
       1. snap
       2. example_domain
     Which domain? Enter a number or domain name:
     :::
     Await the user's response and use it as `<domain>`. Then continue.

2. **Domain folder exists?**
   - NO →
     :::error
     Domain not found: $DOMAINS_DIR/<domain>/
     :::
     Then stop.

3. **Input docs present?**
   - `$DOMAINS_DIR/<domain>/input/policy_docs/` missing or contains no `.md` files →
     :::error
     No input documents found. Add .md files to $DOMAINS_DIR/<domain>/input/policy_docs/ and re-run.
     :::
     Then stop.

## Pre-step A: Sync compressed copies

Run `/compress-inputs <domain>`. Skip pre-flight — domain already verified above.

This pre-step refreshes `policy_facets/compressed/` so it mirrors `input/policy_docs/`. The compressed copies are consumed by downstream skills (`/extract-ruleset`, `/update-ruleset`).

If `/compress-inputs` exits non-zero (per-file caveman failure), abort `/index-inputs` immediately with `:::error` rather than indexing against partial state. The analyst can re-run after addressing the offending file.

## Pre-step B: Plan computations work

Run:

```bash
xlator extract-computations <domain> --plan
```

Parse the JSON output. The shape is:

```json
{
  "to_extract":  [ {"src": "...", "dst": "...", "source_sha": "..."}, ... ],
  "to_delete":   [ "policy_facets/computations/<rel>.md", ... ],
  "noop":        [ {"src": "...", "reason": "unchanged"}, ... ],
  "skipped":     [ {"src": "...", "reason": "not_allowed"}, ... ],
  "succeeded":   [],
  "failed":      []
}
```

This step computes the per-file work plan. The actual per-file AI generation runs in CREATE Step 5 / UPDATE Step 5. The matching `--finalize` runs after the AI loop completes.

`/index-inputs` itself continues to scan `input/policy_docs/` for SHA and md_quality scoring — its index reflects the source files, not the compressed copies and not the computations files.

## Mode Detection

```bash
ls $DOMAINS_DIR/<domain>/policy_facets/input-index.yaml 2>/dev/null
```

| Result | Mode |
|--------|------|
| File absent | **CREATE** — index all files from scratch |
| File present | **UPDATE** — re-index changed/new files only |

## Quality Threshold

The minimum acceptable quality score is **40** (out of 100). Files scoring below this threshold are **REJECTED**: they are moved to `$DOMAINS_DIR/<domain>/input/rejected/` (preserving subdirectory structure) and no per-file computations file is generated for them. The `md_quality:` field in the `files:` block explains the rejection.

---

## Process — CREATE Mode

### Step 1: Discover all input documents

Glob all `.md` files recursively under `$DOMAINS_DIR/<domain>/input/policy_docs/`. Sort alphabetically by path.

:::progress
Found <N> document(s). Indexing all...
:::

### Step 2: Get git SHA and quality score for each file

For each file, run:

```bash
git log -1 --format="%H" -- $DOMAINS_DIR/<domain>/<domain-relative-path>
```

If the output is empty (file is untracked or staged but not committed), use `"untracked"`.

If any files have `"untracked"` SHA, prompt the user before proceeding:

:::user_input
⚠ <N> file(s) are not yet committed to git:
    input/policy_docs/<file1>.md
    input/policy_docs/<file2>.md
  Untracked files are re-indexed on every UPDATE run (no SHA to compare).
  Commit these files now? [y/n]
:::

- **y** → Run:
  ```bash
  git add $DOMAINS_DIR/<domain>/input/policy_docs/<file1>.md $DOMAINS_DIR/<domain>/input/policy_docs/<file2>.md
  git commit -m "chore(<domain>): add input policy docs"
  ```
  Then re-fetch the SHA for each committed file (`git log -1 --format="%H" -- ...`) and use the real SHA in the index.
- **n** → Continue. These files get `"untracked"` SHA and will be re-indexed on every future UPDATE run.

After obtaining each file's SHA, score it:

```bash
xlator rate-md-quality "$DOMAINS_DIR/<domain>/<domain-relative-path>"
```

Store the JSON result `{"score": N}` or `{"score": N, "flags": [...]}` alongside the SHA.

### Step 3: Write the files block to `input-index.yaml`

Ensure `$DOMAINS_DIR/<domain>/policy_facets/` exists (Pre-step B's `--plan` already created `policy_facets/computations/`). Write the following to `$DOMAINS_DIR/<domain>/policy_facets/input-index.yaml` (this creates or overwrites the file):

```yaml
# Auto-generated by /index-inputs — do not edit manually
# Domain: <domain>
# Generated: YYYY-MM-DD
files:
  input/policy_docs/<file1>.md:
    sha: "<sha>"
    md_quality:
      score: <N>
      flags: [<flag1>, <flag2>]
  input/policy_docs/<file2>.md:
    sha: "<sha>"
    md_quality:
      score: <N>
  "input/policy_docs/<file with spaces>.md":
    sha: "<sha>"
    md_quality:
      score: <N>
```

YAML conventions for the `files:` block:
- Two-space indentation throughout
- Quote any path key that contains spaces (e.g., `"input/policy_docs/441-1 EARNED INCOME.md":`)
- `sha:` values as quoted strings
- `md_quality:` sub-block with `score:` as an unquoted integer
- `flags:` as an inline bracket list (e.g., `flags: [no_headings, repeated_page_headers]`) — **omit the `flags:` line entirely** when there are no flags (do not emit `flags: []`)

**Important:** Write each file's entry to the `files:` block one at a time (not all files in a single output block) to avoid hitting the 32k output token limit.

### Step 4: Move rejected files

Run the rejection script, passing the domain directory and the quality threshold:

```bash
xlator reject-low-quality-inputs "$DOMAINS_DIR/<domain>" 40
```

The script reads the `files:` block just written, moves every file whose `md_quality.score` is below 40 from `input/policy_docs/` to `input/rejected/` (preserving subdirectory structure), and returns a JSON result:

```json
{"moved": <R>, "files": ["input/policy_docs/<file>.md", ...]}
```

Capture `<R>` (the moved count) for the completion message.

### Step 5: Generate per-file computations files

For each non-rejected file in sorted order:

- **REJECTED** (score < 40, just moved to `input/rejected/`): Skip entirely — do not invoke `/extract-computations`. Emit a progress line:
  :::progress
    Skipped (low quality, score=<N>): input/policy_docs/<file>.md
  :::

- **All other files**: Invoke the per-file skill:

  ```
  /extract-computations <DOMAINS_DIR>/<domain>/input/policy_docs/<rel>.md
  ```

  Skip pre-flight inside `/extract-computations` — domain and source were already validated above.

  On success, append the source path (`input/policy_docs/<rel>.md`) to the `succeeded:` list inside `<DOMAINS_DIR>/<domain>/policy_facets/.extract-plan.tmp` (the file Pre-step B wrote). Save the file after each update so a crashed run can be recovered by `--finalize`.

  On failure, append `{"src": "<rel>", "error": "<short message>"}` to the `failed:` list and continue with the next file. Do NOT abort the whole CREATE run on one bad file.

  Emit a progress line:
  :::progress
    Extracted: input/policy_docs/<file>.md (<K> section(s))
  :::

### Step 6: Finalize computations work

Run:

```bash
xlator extract-computations <domain> --finalize
```

The tool:
- deletes destination copies for any `to_extract` entry not in `succeeded:` (so the next run reattempts them),
- applies mirror-deletes from `to_delete` and prunes their manifest entries,
- writes the manifest atomically (`tmp + os.replace`),
- removes `.extract-plan.tmp`.

Relay its summary inside `:::detail`:

:::detail
<verbatim output of the finalize command>
:::

If the `failed:` list is non-empty, emit:

:::important
⚠ <N> file(s) failed to extract:
  - <src1>: <error1>
  - <src2>: <error2>
The corresponding computations files were removed; re-running /index-inputs will retry.
:::

Exit non-zero in this case so any caller can decide whether to proceed.

### Step 7: Completion

After all files are processed, print:

:::important
✓ policy_facets/input-index.yaml written (files block).
  policy_facets/computations/ written (<N> file(s) extracted).
  <R> file(s) rejected (low quality) and moved to input/rejected/ — re-run after reformatting.
:::

Omit the "rejected" line if R = 0.

:::next_step
Next step: `/refine-guidance <domain>` to set extraction goals and ruleset guidance
:::

## Process — UPDATE Mode

### Step 1: Read the existing files block

Read `$DOMAINS_DIR/<domain>/policy_facets/input-index.yaml`. Extract the current `files:` SHA map (including any existing `md_quality:` values).

### Step 2: Discover current input documents

Glob all `.md` files recursively under `$DOMAINS_DIR/<domain>/input/policy_docs/`. Sort alphabetically by path.

### Step 3: Get current git SHA and quality score for each file

For each current file, run:

```bash
git log -1 --format="%H" -- $DOMAINS_DIR/<domain>/<domain-relative-path>
```

Empty output → `"untracked"`.

For files that will be classified as **REINDEX** (see Step 4), also score them now:

```bash
xlator rate-md-quality "$DOMAINS_DIR/<domain>/<domain-relative-path>"
```

Store the JSON result alongside the SHA.

For files that will be classified as **SKIP** (SHA unchanged), use the `md_quality:` value already stored in the prior index. If the prior index has no `md_quality:` for a SKIP file (e.g., it was indexed before this feature was added), score it now using the same command.

### Step 4: Classify files

Compare current files + SHAs against the stored `files:` block:

| Condition | Classification |
|-----------|----------------|
| File in index, SHA unchanged | **SKIP** — preserve existing per-file computations file and md_quality |
| File in index, SHA changed | **REINDEX** — re-extract via `/extract-computations` |
| File not in index (new) | **REINDEX** — extract via `/extract-computations` |
| REINDEX file with score < 40 | **REJECTED** — move to `input/rejected/`; no computations file |
| File in index but not on disk (deleted) | **REMOVE** — drop from `files:`; the computations manifest's `to_delete` already handled the mirror-delete in Pre-step B |

:::progress
Found <N> document(s): <X> changed/new, <Y> unchanged, <Z> deleted.
Re-indexing <X> file(s)...
:::

If any current files have `"untracked"` SHA, prompt the user (same flow as CREATE Step 2).

### Step 5: Re-extract REINDEX files

For each REINDEX file (in alphabetical path order, REJECTED filtered out):

  ```
  /extract-computations <DOMAINS_DIR>/<domain>/input/policy_docs/<rel>.md
  ```

Skip pre-flight inside `/extract-computations`. Update `succeeded:` / `failed:` in `<DOMAINS_DIR>/<domain>/policy_facets/.extract-plan.tmp` after each file (same shape as CREATE Step 5).

Emit a progress line per file:
:::progress
  Extracted: input/policy_docs/<file>.md (<K> section(s))
:::

REJECTED files (score < 40) are skipped — they produce no computations file. SKIP files are not invoked — Pre-step B's manifest classification already routed them as `noop`, and their existing per-file file remains untouched.

### Step 6: Write the updated `input-index.yaml` (files block only)

Write the following to `$DOMAINS_DIR/<domain>/policy_facets/input-index.yaml` (overwrites the existing file):

```yaml
# Auto-generated by /index-inputs — do not edit manually
# Domain: <domain>
# Generated: YYYY-MM-DD
files:
  <all current files with their current SHAs and md_quality>
```

Apply the same YAML conventions as CREATE mode Step 3:
- Quote paths with spaces; `sha:` as quoted strings; `md_quality:` sub-block with unquoted `score:`; `flags:` omitted when empty
- Write each file's entry one at a time (incremental write)

### Step 7: Move rejected files

Run the same rejection script used in CREATE mode:

```bash
xlator reject-low-quality-inputs "$DOMAINS_DIR/<domain>" 40
```

Capture `<R>` from the JSON result for the completion message.

### Step 8: Finalize computations work

Run:

```bash
xlator extract-computations <domain> --finalize
```

Relay the summary inside `:::detail`. If `failed:` is non-empty, emit a `:::important` block listing the failures (same shape as CREATE Step 6).

For each REJECTED file, emit:
:::progress
  Skipped (low quality, score=<N>): input/policy_docs/<file>.md
:::

### Step 9: Completion

After all files are processed, print:

:::important
✓ policy_facets/input-index.yaml updated (files block).
  policy_facets/computations/ updated (<X> file(s) re-extracted, <Y> skipped, <Z> removed).
  <R> file(s) rejected (low quality) and moved to input/rejected/ — re-run after reformatting.
:::

Omit the "rejected" line if R = 0.

:::next_step
Next step: `/refine-guidance <domain>` to set extraction goals and ruleset guidance
:::

## Common Mistakes to Avoid

- **Don't write `policy_facets/input-sections.yaml`** — that artifact is removed in v3.0.0. Per-file section data lives at `policy_facets/computations/<rel>.md`. Pre-existing `input-sections.yaml` files (from prior versions) are ignored by the new flow; maintainers delete them manually.
- **Don't manually edit `policy_facets/computations/<rel>.md`** — they are derived by `/extract-computations`. Edit the source under `input/policy_docs/` and re-run `/index-inputs`.
- **Don't skip `xlator extract-computations --finalize`** — without it, the manifest is not updated and the next run re-extracts every file.
- **Don't use absolute paths in the index** — `files:` keys must start with `input/policy_docs/`, not `/Users/...` or `$DOMAINS_DIR/<domain>/`.
- **Don't run the rejection script before the files block is fully written** — `reject-low-quality-inputs` reads `policy_facets/input-index.yaml` for scores; run it only after CREATE Step 3 / UPDATE Step 6 is complete.
- **Don't re-extract unchanged files in UPDATE mode** — only invoke `/extract-computations` for REINDEX files. Pre-step B's manifest already classifies SKIP files as `noop`; touching their per-file files burns AI tokens for no gain.
- **Do quote `files:` keys that contain spaces** — YAML keys with spaces must be quoted strings.
- **Don't emit `flags: []` for files with no quality flags** — omit the `flags:` line entirely; `score:` alone is sufficient.
- **Don't skip md_quality scoring** — every file in the `files:` block must have an `md_quality:` entry.
- **Don't invoke `/extract-computations` for REJECTED files** — they're moved to `input/rejected/` before Step 5; no per-file file is generated for them.
- **Don't abort on first per-file `/extract-computations` failure** — append to the `failed:` list and continue. After the loop, surface the failures via `:::important` and exit non-zero so the analyst can address them.
