# Specifications

This directory contains shared specification files used by domain-specific specifications extracted from the input docs and code from the legacy system. These specifications are used to generate the output ruleset and code for the modernized system.

The specifications capture government policies and statutes, including:
- eligibility rules
- government benefits
- tax regulations
- etc.

In addition to decision logic, the specifications cite source documents/code for traceability.

## Source format: Catala

The authored source format is [Catala](https://catala-lang.org) (`.catala_en` literate-Markdown). The AI emits Catala directly under `domains/<d>/specs/<module>.catala_en`; SMEs read the Catala source alongside its literate-Markdown structure (`## Heading` sections mirror `policy_facets/computations/<rel>.md.yaml`, with inline `*Source: ...*` italic-prose citations).

### Key features

- **First-class exceptions and prioritized defaults.** Catala's `definition` / `under condition` / `exception` constructs map directly to legal-policy specification patterns; the AI does not need to encode them in a custom DSL.
- **Module composition.** `> Using <Module>` directives compose cross-module type contracts; `clerk typecheck` enforces them.
- **Multi-target output (structural).** Catala compiles to C, Java, OCaml, Python, JS (via OCaml→JS), and DOCX explanations via [`catala-explain`](https://github.com/CatalaLang/catala-explain). Building specific backends is out of scope for this project; the option remains for downstream consumers.

### Use cases

- Government benefit eligibility rules
- Tax filing requirements and instructions
- Complex multi-jurisdiction policy scenarios
- Rules requiring full audit trails with legal citations

## Testing

All rule specifications should have corresponding test suites:
- Tests should cover:
  - Happy path scenarios
  - Edge cases and boundary conditions
  - Invalid inputs
  - Policy reference validation

The YAML test format (`specs/tests/<module>_tests.yaml`) is preserved; `xlator catala-test-transpile` lowers it to `#[test]` scopes against the Catala source.

### Generated Files — Do Not Edit by Hand

The following files are **generated by build scripts** and must not be edited manually. Re-run the appropriate `./xlator` command to regenerate after changing a Catala source or test YAML.

| Generated file | Source | Command |
|----------------|--------|---------|
| `$DOMAINS_DIR/<domain>/output/<module>.catala_en` | `specs/<module>.catala_en` | `./xlator catala-pipeline <domain> <module>` (copies source to output) |
| `$DOMAINS_DIR/<domain>/output/tests/<module>_tests.catala_en` | `specs/tests/<module>_tests.yaml` + `specs/naming-manifest.yaml` | `./xlator catala-test-transpile <domain> <module>` |

**Full pipeline** (typecheck → test transpile → test in one command):

| Command | What it runs |
|---------|-------------|
| `./xlator catala-pipeline <domain> <module>` | `clerk typecheck` → `catala-test-transpile` → `clerk test` |

**Authoring workflow:** Edit `specs/<module>.catala_en` directly (or `specs/tests/<module>_tests.yaml`), then re-run the pipeline. Never edit `.catala_en` files under `output/` — they are copies of the source maintained by the build step.

## Principles

1. **Deterministic**: Same inputs + rules → same outputs
2. **Traceable**: Every decision has clear reasoning
3. **Cited**: Rules reference authoritative policy sources
4. **Versioned**: Rules have effective dates and version tracking
5. **Testable**: All rules have comprehensive test coverage
6. **Explainable**: Decisions can be explained to end users

## When adding new specifications

1. Author the Catala source under `domains/<d>/specs/<module>.catala_en`
2. Always include corresponding test cases
3. Document policy references and citations (literate-Markdown headings + `*Source: ...*` italic-prose lines)
4. Include effective dates and version information
5. Ensure rules are deterministic and explainable
